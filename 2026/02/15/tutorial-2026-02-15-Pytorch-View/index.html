Pytorch View

pytorch的各类操作返回的是视图还是副本？其对张量的操作是如何高效完成的？
可以从底层存储的角度进行分析与澄清，还是很好理解的

视图view

def: 视图 是一个与原始张量共享相同底层数据的新张量。它拥有自己的形状(size)、步长(stride) 等元信息，但不独立拥有数据。修改视图中的元素会直接影响原始张量，反之亦然。

可以理解为c++中的引用

返回视图的操作是零开销的，不需要开辟新空间和搬运开销，很高效。

修改视图时，指向统一存储空间的原始张量和其他视图也会相应被修改。

stride与连续性

stride: 步长。tensor.stride(d)表示该张量在dim=d维度上的步长，即沿着这个维度上走，下一个元素相对于自己这个元素的存储地址的偏移量。

  • 对于一个 shape=(d0,d1,...,dn−1),stride=(s0,s1,...,sn−1)shape = (d_0, d_1, ..., d_{n-1}), \quad stride = (s_0, s_1, ..., s_{n-1})shape=(d0​,d1​,...,dn−1​),stride=(s0​,s1​,...,sn−1​) 的张量，满足索引为 (i0,i1,...,in−1)(i_0, i_1, ..., i_{n-1})(i0​,i1​,...,in−1​) 的元素在内存中相对张量基地址的偏移为 i0∗s0+i1∗s1+...+in−1∗sn−1i_0*s_0 + i_1*s_1 + ... + i_{n-1}*s_{n-1}i0​∗s0​+i1​∗s1​+...+in−1​∗sn−1​
    stride定义了张量是如何诠释自己所指向的存储空间的，也因此可以零开销实现视图操作

用tensor.is_contiguous()可以判断该张量(不论是原始张量还是视图)是否是连续存储的
这里连续的定义是这个张量是行优先存储的，满足条件：若shape=(d0,...,dn−1)shape=(d_0,...,d_{n-1})shape=(d0​,...,dn−1​)，则 stridei=Πk=i+1n−1dkstride_i=\Pi_{k=i+1}^{n-1}d_kstridei​=Πk=i+1n−1​dk​，即stride单调下降，且striden−1=1stride_{n-1}=1striden−1​=1

view & reshape

  • view只能作用在连续存储的tensor，返回视图；否则报错
  • reshape在原有张量是连续的情况下，返回的是原始张量的视图(与view的行为基本一致)
    reshape在原有张量不连续的情况下，会先创建一个副本，其为连续存储且与原有张量逻辑上是一致的，然后再在这个副本上进行与view类似的操作。返回的是新的张量而非原始张量的视图

显然，view和reshape返回的张量(无论是视图还是副本)，都是连续的

下面是对a=torch.arrange(0,12)进行各种reshape的结果

可以看出，不论怎么reshape，对其中任意一个元素，沿着最后一个维度的下一个元素都是一致的，把每个张量写成一行都是0到11。因为存储空间中的数据都没有变化，而且最后一个stride总是1

索引、permute、transpose

索引

  • 基本索引：由:、整数、None、...构成的基本索引返回的是原始张量的视图。但可能连续可能不连续，取决于具体情况 类比：squeeze()、unsqueeze()、expand()是插入或删除一个长度为1的维度，无论输入张量是否连续，返回的都是视图。
  • 高级索引：只要a[<index_0>,<index_1>,...]中的至少一个<index_i>不是构成基本索引的三种符号之一，那么返回的就是视图
    这个也很好理解，因为我们无法通过修改stride就能在原有的地址空间上构建出高级索引想要得到的张量（在同一个维度上，stride都是相等的）。

permute

permute返回的总是视图，但一般不连续（除非permute回了和原始张量一致的情况）

permute在底层上就是交换对应维度的stride数值

transpose

交换两个维度的stride，其实本质上与permute是一致的

计算操作

  • in-place operation: 直接修改原有张量而不创建一个新的拷贝副本
    例如tensor.add_(42)，在torch中，在最后带有_的成员函数都是就地操作
    因为torch对就地操作比较敏感，因为它会破坏自动求导的计算图，所以最好使用其成员函数而非tensor += 42
    当然，这个操作不会改变张量的任何性质。但是和它指向同一个存储空间的所有视图和原始张量都会随之改变。
    所以慎用就地操作，一般在自己手动实现autograd函数对象的前传和反传的时候才会这么干，以节约显存占用和复制开销
  • 其他计算操作：t + 1, torch.sin(t)都会创建一个副本，保存输入张量经过计算后的结果。当然输入张量本身不会发生改变。

Attention: 这意味着，t += 1 和 t = t + 1 并不等价
t += 1是就地操作，会破坏torch创建的计算图使其无法自动求导
t = t + 1是创建了一个副本保存+1后的副本张量，然后把标识符t绑定到了副本张量上。原来的t绑定的张量并不会被丢弃，而是会被torch记住，用于自动求导和反传
